# 字幕翻訳オーバーレイ設計書

## 実装ステータス
> **✅ 実装完了** (2026-03-11)
> YouTube 字幕の取得強化、Gemini / Z.AI 翻訳、構造化出力、ステータス UI の小型化まで実装済みです。

## 1. 概要

### 1.1 目的

YouTube の文字起こしを動画上へオーバーレイ表示しつつ、Gemini または Z.AI を使った逐次翻訳を重ねて表示できるようにします。字幕取得は YouTube 側の UI や API 変化に左右されやすいため、複数の取得経路を持たせて復元力を高めます。

### 1.2 スコープ

- YouTube 動画ページでの字幕オーバーレイ表示
- 歯車から 1〜5 行で変更できる字幕集約表示（初期値 5 行）
- 3 ブロック単位の翻訳ウィンドウ
- Gemini / Z.AI への翻訳リクエスト
- 翻訳中は原文をフォールバック表示
- 日本語の自動生成字幕は翻訳対象から除外
- ステータス表示の小型化とエラー可視化

### 1.3 設計原則

- `KISS`: 表示単位は歯車から変えられる 1 項目だけに絞り、翻訳単位は 3 ブロック固定のままにして挙動の単純さを優先しています。
- `YAGNI`: 対象言語は日本語固定、翻訳ジョブは逐次実行のみ、プロバイダ切替は Gemini / Z.AI の 2 つに絞っています。
- `DRY`: 字幕の正規化、翻訳ウィンドウ管理、ステータス描画は共通関数に寄せています。

## 2. アーキテクチャ

### 2.1 ファイル構成

- `content.js`
  - YouTube DOM 上での字幕取得
  - オーバーレイ UI 描画
  - 再生同期
  - 翻訳キュー管理
  - provider 切替 UI と provider 別設定保持
- `background.js`
  - Gemini / Z.AI API 呼び出し
  - Gemini structured output の schema 指定
  - リトライ制御
- `page-bridge.js`
  - ページコンテキストから `playerResponse` を取得
- `manifest.json`
  - MV3 service worker と API host permission の定義

### 2.2 データフロー

1. `content.js` が現在の動画 ID を検出します。
2. 字幕取得を以下の優先順で試します。
   - `youtubei/v1/get_transcript`
   - `playerResponse.captionTracks`
   - `ANDROID` / `IOS` client の `youtubei/v1/player`
3. 取得した字幕行を歯車設定の `字幕連結数` 単位で `groupedSegments` に変換します。
4. `groupedSegments` を 3 ブロック単位の翻訳ウィンドウに切ります。
5. `background.js` が選択中 provider へ翻訳リクエストを送り、JSON 配列で翻訳結果を返します。
6. `content.js` が index と件数を検証し、`translatedText` として反映します。
7. 再生位置に応じて `translatedText ?? text` を動画上へ表示します。

## 3. 字幕取得設計

### 3.1 多段フォールバック

YouTube の字幕取得は動画ごとに挙動が異なるため、単一経路に依存しません。

- `get_transcript` が有効な場合は API 応答を使います。
- `WEB` client の `captionTracks` が空本文になる動画があるため、必要時は `ANDROID` と `IOS` の `player` endpoint から字幕トラックを取り直します。

### 3.2 採用理由

- `KISS`: YouTube UI を直接操作する panel fallback は、画面フォーカス移動など副作用のわりに成果がなかったため廃止しました。
- API は UI 操作より副作用が少なく、拡張側で完結できます。
- モバイル client fallback は、`WEB` では空でも字幕本文を返す動画があるため採用しています。

## 4. 翻訳設計

### 4.1 翻訳単位

- 表示単位: 歯車設定で可変（初期値 5 行）
- 翻訳単位: 3 ブロック

翻訳単位を 20 から 3 に縮めた理由は以下です。

- 初回レスポンスを速くするため
- モデルが複数字幕を結合しやすい問題の影響範囲を小さくするため
- シーク時の追従性を上げるため

### 4.2 実行方式

- 並列ではなく逐次実行
- 現在再生位置の未翻訳ウィンドウを優先
- 翻訳失敗時も原文表示は継続
- 字幕トラックが `日本語（自動生成）` の場合は翻訳キューを作らず、そのまま原文表示を続行

### 4.3 Provider ごとの応答形式

Gemini には schema 付き JSON 配列を返させます。Z.AI には chat completions 互換 endpoint へ JSON 配列のみ返すよう指示します。

返却形:

```json
[
  { "index": 1, "text": "..." },
  { "index": 2, "text": "..." }
]
```

意図:

- SRT 区切り崩れによる件数ズレを減らす
- index 検証で並び替えや欠落を検知する
- timestamp は元データをそのまま使い、モデルには翻訳文だけを担当させる

### 4.4 リトライ方針

- 最大 5 回リトライ
- 一時的失敗だけでなく構造化応答の揺れも再試行対象
- 最終失敗時は対象ウィンドウを `error` にし、他のウィンドウ処理は継続

### 4.5 Provider 切替

- UI から `Gemini` / `Z.AI` を選択できます。
- API Key とモデル名は provider ごとに別保存します。
- `KISS`: provider ごとの差は background.js に閉じ込め、content.js 側は共通の翻訳キューをそのまま使います。
- `DRY`: provider ごとの設定解決は共通 helper でまとめ、保存・表示・送信で同じ分岐を繰り返さないようにしています。

## 5. UI 設計

### 5.1 字幕オーバーレイ

- 動画上に中央下寄せで表示
- ドラッグ移動可
- ダブルクリックで位置リセット

### 5.2 ステータス表示

- 通常は左上の小さいアイコンだけ表示
- 読み込み開始・読み込み完了・エラー時のみ自動展開
- hover で詳細文を表示
- エラー時は赤アイコン
- 本文は選択してコピー可能
- アイコン自体はドラッグ移動可能

## 6. エラー処理

### 6.1 字幕取得

- 各取得経路の失敗理由を連結して表示
- 最後の一手まで失敗したときだけ致命扱い

### 6.2 翻訳

- JSON 件数不一致
- index 不一致
- 空行過多
- Gemini / Z.AI API エラー

上記はすべてウィンドウ単位の失敗として扱い、アプリ全体は止めません。
